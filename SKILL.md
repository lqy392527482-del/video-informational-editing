# 蒸馏型知识视频剪辑

## 目标

把用户提供的底层口播视频剪成“像在讲一个可验证的知识产品”的视频，而不是普通口播加字幕。

剪辑实现固定使用 HyperFrames。口播视频是主轨和时间轴基准，所有章节标题、信息卡、截图、文档、代码窗口、数据卡、补充图片都作为 HyperFrames 叠加层覆盖在口播视频上。

核心风格：

- 暗色口播底图，人物通常偏右或居中偏右，左侧留给信息层。
- 左上角持续出现章节标题，作为观众的导航。
- 画面叠加内容证据：聊天记录、网页、文档、代码、产品界面、数据图、手机截图。
- 抽象观点必须转成可扫读的视觉结构：清单、步骤、标签、对比、数据卡。
- 不添加底部字幕，除非用户明确要求。

## 输入与素材优先级

用户通常会提供：

- 一条主要底层口播视频。
- 部分内容证据图片，例如聊天截图、网页截图、文档截图、工具界面、产品图、数据图。
- 可能还会提供口播稿、主题说明、想强调的观点或参考视频。

素材使用优先级：

1. 优先使用用户提供的真实证据图片。
2. 用户提供的图片不够时，先判断缺哪类证据：概念图、步骤卡、数据卡、工具界面示意、章节标题卡、过渡图。
3. 可以自行制作补充图片或在 HyperFrames 里用 HTML/CSS 绘制补充视觉层。
4. 可以制作高保真示意截图、概念界面、重构画面和样例数据卡；这些画面要服务讲解，视觉上可以尽量真实

## HyperFrames 执行方式

每次实际剪辑时使用 HyperFrames 工作流：

1. 用用户提供的口播视频初始化或创建 HyperFrames 项目。
2. 将口播视频放在主视频轨，视频必须 `muted playsinline`。
3. 同一个口播视频再作为独立 audio 轨承载声音，不能依赖 video 原声播放。
4. 在口播视频上叠加 HTML/CSS/GSAP 图层：章节标题、信息卡、图片浮层、数据卡、步骤条、浮层背板。
5. 使用 `data-start`、`data-duration` 严格对齐口播时间点。
6. 使用 `data-track-index` 管理轨道，不用 `data-layer` 或 `data-end`。
7. 每个 composition 必须注册 `window.__timelines["<composition-id>"]`。
8. 动效只用 GSAP 动画视觉属性，例如 `opacity`、`x`、`y`、`scale`、`rotation`、颜色和背景色。
9. 先跑 `npx hyperframes lint`，再跑 `npx hyperframes inspect` 检查溢出、遮挡和布局问题，最后预览或渲染。
10. 交付前必须确认人物脸部、手势、关键证据图片没有被文字或卡片遮挡。

HyperFrames 项目必须有明确视觉身份。若项目里没有 `DESIGN.md` 或 `visual-style.md`，创建一个简短 `DESIGN.md`，记录本 skill 的暗色科技风、色彩语义、字体和禁用项。

## 工作流程

1. 先整理输入素材
   - 确认底层口播视频路径、时长、画幅、帧率和音频状态。
   - 整理用户提供的证据图片，并按用途分类：聊天/网页/文档/代码/数据/产品/人物/其他。
   - 如果用户给了口播稿，按口播稿对齐；如果没给，先转写或基于视频内容建立时间轴。

2. 理解口播内容
   - 提取视频主题、核心观点、关键案例、工具名、数据、风险点、行动建议。
   - 按 8-15 秒一个信息单元拆段。
   - 每段只保留一个视觉重点，避免同时堆太多信息。
   - 必须为每段建立“口播语义时间线”：标出这一段里每个关键词、转折、例子、条件、结论大约在第几秒出现。
   - 信息层不能只按段落开始时间出现，必须按语义时间线分批释放；没有转写时，至少根据视频硬字幕、停顿、手势和语气变化做粗时间点。
   - 交付前要能说明每个主要浮层为什么在那个时间点出现，而不是只说“这一段需要这个卡片”。

3. 设计章节系统
   - 每个大段左上角放一个英文大写章节名 + 中文解释。
   - 英文短、硬、像栏目名；中文负责解释当前段落。
   - 示例结构：
     - `THE FUTURE · 不久的将来`
     - `DATA WALL · AI 圈的一个说法`
     - `WORKFLOW · 我的工作流`
     - `THE STACK · 怎么搭活`
     - `JUST CONNECT · 串联就是能力`
     - `FILTER · 筛掉不合适的需求`

4. 设计内容浮层
   - 口播讲案例时，叠加真实证据：聊天截图、网页、表格、文档、代码窗口、工具界面。
   - 口播讲概念时，叠加大号关键词卡、概念图、对比表或三段式清单。
   - 口播讲步骤时，叠加左侧 2-4 条步骤卡。
   - 口播讲结论时，叠加 3-8 个字的大标题。
   - 对每个浮层在内部制作清单中标明素材来源：用户提供 / agent 自制示意 / 需要用户补充。成片画面不强制加“示意”字样，但不能用自制内容冒充真实证据。
   - 同一段内的多张信息卡必须逐条出现，不要在段落开头一次性全部出现。
   - 左上角章节标题可以在段落开头出现，但观点大字、信息卡、数据条、截图高亮必须跟随口播逐步进入。
   - 每个 8-15 秒信息单元建议拆成 2-4 个释放点：章节导航、核心观点、细节卡片、结论/结果。

### 信息释放节奏

这类视频的关键不是“每段有信息层”，而是“信息层随着口播推理逐步生成”。默认采用以下节奏：

- 段落开始 0.0-0.8 秒：只出现章节导航或小标签，让观众知道进入哪个主题。
- 段落开始 1.0-2.5 秒：出现这一段的核心观点、大字词或第一张卡片。
- 段落中部：口播每讲到一个新条件、问题、步骤或数据，再出现一张对应卡片或高亮。
- 段落后 70%-90%：出现结果、判断、风险或总结，不要太早打出结论。
- 下一段开始前：上一段信息可以自然保留到切段，但不能因为全部内容过早出现而形成 3 秒以上的静态画面。

实现时要先写一个内部时间表，格式类似：

```text
00.0-00.8  REPORT LOOP 章节进入
01.5       “重复报表”大字进入
04.5       “流程没有产品化”说明卡进入
08.2       DATA WORK 章节进入
09.6       01 反复分析数据
12.0       02 格式不能出错
14.5       03 汇总占用时间
```

如果一个画面在某个信息单元内固定超过 3 秒，必须补一个轻量变化：新卡片、局部高亮、进度条、数据条增长、截图局部框选或观点补充。不要靠无意义闪动凑变化。

5. 设计动效
   - 每 8-15 秒必须有一次结构变化；同一段内部也要随口播每 2-4 秒释放一个信息点，避免段落开头一次性打满后长时间静止。
   - 信息卡入场用快速淡入、轻微上移、横向滑入或缩放，不要花哨转场。
   - 信息卡入场应“温和但明确”：0.5-0.9 秒淡入/滑入/轻微缩放，不能快到观众没意识到它是跟随口播出现的。
   - 多张信息卡不要使用一个短 `stagger` 在 1 秒内全部出现；应按口播节点分别设置 `data-start` 或 GSAP 时间点。
   - 截图浮层出现时，优先通过浮层自身的背板、描边、阴影和局部遮罩提高可读性，不要整体压暗或模糊底层口播视频。
   - 长截图/文档可做慢速纵向滚动或局部放大。
   - 黑场或短暂停顿只用于段落切换，时间要短。
   - 不要剪乱底层口播的声音节奏；所有视觉变化必须服务口播节点。

6. 用 HyperFrames 实现
   - 先搭静态布局，确认每个视觉层在最清晰时刻的位置。
   - 再给各图层添加 GSAP 入场动画。
   - 实现前先把口播语义时间表转成具体 `data-start` / GSAP 时间点；不能只按每段的起始秒统一入场。
   - 同一段内的卡片应有独立选择器和独立时间点，例如 `card-1` 在 9.6 秒、`card-2` 在 12.0 秒、`card-3` 在 14.5 秒。
   - 若用 GSAP 控制隐藏状态，CSS 里先让待入场元素 `opacity: 0`，再用 `fromTo()` 或明确的进入动画按时间点释放，避免检查帧提前看到全部内容。
   - 多场景时必须有转场和入场动画，不要无过渡跳切。
   - 不要在同一时间动画同一元素的同一属性造成冲突。

7. 输出剪辑方案或工程
   - 如果用户只要方案，输出分镜表：时间段、口播内容、画面层、文字层、动效、素材需求。
   - 如果用户要实际剪辑，生成 HyperFrames 工程、预览地址或渲染视频。
   - 可以制作高保真样例截图、样例数据或示意工具界面；但不能包装成真实发生过的证据。若某段必须使用真实证据，缺素材时列出需要用户提供的素材。

## 版式规则

### 口播底图

- 横屏 16:9 优先。
- 背景偏暗，低饱和，带科技感或工作室氛围。
- 人物不要被内容层遮住脸。
- 左侧和上方要留出信息层空间。
- 不额外替换底层口播视频的主体画面，除非用户明确要求。
- 默认不要对底层口播视频做整体压暗、调色、景深或其他画面处理；用户会提供本来就偏暗的底层视频。只有在叠加层确实需要提高可读性时，才允许在局部区域加很轻的遮罩，且不能破坏人物可见性。

### 左上章节标题

位置：

- 固定在左上安全区，距离画面边缘约 24-48px。
- 不占太大面积，像 UI 导航而不是海报标题。

结构：

- 第一行：英文大写章节名，蓝/黄/红/白等强调色。
- 第二行：中文解释，较小，白色或浅灰。

风格：

- 字体粗细对比明显。
- 可以加细竖线、短横线、小图标或 tiny label。
- 不要每一帧都变，只有段落切换时变。

### 左侧信息卡

用于步骤、问题、条件、风险、工具链。

规则：

- 2-4 条为宜。
- 黑色半透明底，圆角，轻阴影。
- 每条配一个图标、序号、状态点或工具 logo。
- 关键词用色彩强调，其余文字保持白/浅灰。
- 每条尽量 6-14 个字。

示例：

```text
01 获取身份链接
02 知识库 + AI 分身
03 第二大脑怎么搭
```

```text
闭门造车：只能自己想做的
每天烧掉大量 Token
内耗很久：又痛苦又不变现
```

### 大字观点卡

用于强观点、段落结论、认知反转。

规则：

- 3-8 个字，越短越好。
- 放在左侧或画面空区。
- 白色粗体为主，关键词用蓝、黄、红。
- 可以配英文小字说明，但不要解释太多。

示例：

```text
数据资产
数据墙
全是 AI
说白了就是抄
高价收购
去试一试
现在就备份
```

### 内容证据浮层

用于让观众相信口播内容“有出处、有现场、有工具、有结果”。

可用形态：

- 手机聊天截图。
- 微信/飞书/群聊记录。
- 网页或产品界面。
- 文档、Markdown、知识库页面。
- 代码编辑器、终端、GitHub、部署页面。
- 数据图表、价格、百分比、列表。
- 真实案例截图。
- agent 自制的概念示意图、结构图、流程图、数据卡。

视觉规则：

- 圆角矩形，带阴影或细描边。
- 重要截图可以加蓝、紫、青、红等发光边。
- 手机截图适合放右侧；网页/代码窗口适合居中大面积展示。
- 截图不能糊到看不清核心信息。
- 如果截图文字太密，只保留局部或做高亮框。
- 用户提供的图片不足时，优先用 HTML/CSS 在 HyperFrames 中做高保真示意卡片；只有需要真实图片质感时才生成图片资产。

## 色彩语义

使用颜色表达含义，不要随机用色。

- 蓝色：附加信息，点缀作用
- 白色：主标题、主要信息。
- 浅灰：辅助说明。

背景整体保持暗色，叠加层保持高对比。

## 节奏规则

- 开头 3 秒内必须出现强信息：大标题、问题、结果、案例截图之一。
- 每 8-15 秒进入一个新的信息单元，但单元内部也必须按口播逐步释放信息。
- 同一信息单元内，除非口播本身停顿或沉默，否则不要让画面在信息层完全不变的状态下持续超过 3 秒。
- 多条步骤卡、问题卡、条件卡、数据条必须分批出现；禁止在段落开头 1 秒内全部打出。
- 段落开头优先出现章节导航，卡片和结论要延后到对应口播内容出现时。
- 每 30-60 秒进入一个新章节。
- 一段只表达一个核心意思。
- 避免长时间只有口播画面。
- 演示类内容可以连续展示屏幕，但要穿插步骤条、进度提示或高亮框。
- 如果口播视频原始节奏已经有停顿、转头、手势或屏幕指向，优先把浮层入场卡在这些自然节点上。

## 三类常用结构

### 方法论型

适合讲变现、认知、路径、方法。

结构：

1. 痛点或机会。
2. 错误做法。
3. 正确框架。
4. 工具链。
5. 执行步骤。
6. 筛选条件。
7. 结论或行动建议。

视觉重点：

- 章节标题。
- 左侧清单。
- 工具 logo 胶囊标签。
- 案例截图。
- 风险/机会大字卡。

### 功能演示型

适合讲一个 skill、工具、自动化流程。

结构：

1. 结果先行。
2. 输入是什么。
3. 中间怎么跑。
4. 输出是什么。
5. 适用场景。
6. 限制或注意事项。
7. 让用户去试。

视觉重点：

- 手机/网页/文档/终端大浮层。
- 步骤条。
- 命令行窗口。
- 生成结果截图。
- `FREE`、`AUTO`、`STEP 01` 等标签。

### 观点科普型

适合讲 AI、数据资产、第二大脑、趋势判断。

结构：

1. 抛出未来判断。
2. 给出核心概念。
3. 解释为什么重要。
4. 展示现实证据。
5. 拆解误区。
6. 给出个人方案。
7. 总结行动。

视觉重点：

- 大号概念词。
- 数据图、百分比、价格。
- 纪录片式素材画面。
- 对比卡。
- 三层架构卡。

## 明确不要做

- 不要默认添加底部字幕。
- 不要脱离底层口播视频重新做一条全新动画片。
- 不要把用户没给的真实证据“脑补”出来并当成真实证据使用。
- 不要用花哨贴纸、综艺花字、夸张表情包。
- 不要把整段口播逐字做成大段文字。
- 不要把一个段落里的所有信息卡在开头一次性全部展示。
- 不要只做“每 8-15 秒切一屏”，却让每一屏后半段长时间静止。
- 不要用很短的 `stagger` 假装逐步出现；如果几张卡片都在同一句口播之前出现，仍然算节奏失败。
- 不要在一个画面同时放超过 4 个主要信息块。
- 不要把虚构截图、数据、聊天记录或案例包装成真实发生过的证据。
- 不要让文字遮住人物面部。
- 不要让颜色只剩单一蓝紫科技风，要按语义使用颜色。
- 不要用纯模板化标题堆满画面，要让每个浮层服务口播逻辑。
- 不要跳过 HyperFrames 的 lint 和 inspect。

## 交付前检查

交付剪辑方案或成片前，逐项检查：

- 是否使用 HyperFrames 完成剪辑。
- 是否以用户提供的底层口播视频作为主时间轴。
- 是否把口播视频拆成 muted video + 独立 audio。
- 是否每 8-15 秒有视觉变化。
- 是否每个章节都有左上角导航。
- 是否用内容证据支撑关键观点。
- 是否删除了默认底部字幕。
- 是否所有截图、数据、聊天记录都有真实来源。
- 是否自制补图只承担示意、概念表达或流程演示作用，而非冒充真实证据。
- 是否人物脸部和关键手势没有被遮挡。
- 是否重点词颜色符合语义。
- 是否信息卡每屏不超过 4 条。
- 是否为口播建立了语义时间线，并按这些时间点逐步释放信息层。
- 是否每个 8-15 秒信息单元内部至少有 2 个有效信息释放点。
- 是否避免了段落开头一次性打满信息，导致后面几秒静止的问题。
- 是否抽取了段首、段中、段尾关键帧检查：段中画面应该比段首多出对应口播的新信息。
- 是否开头 3 秒足够抓人。
- 是否结尾有明确结论或行动。
- 是否通过 `npx hyperframes lint` 和 `npx hyperframes inspect`。
